# RWANG Optimize Policy

Optimize implementation only; preserve approved architecture and public contracts.

1. Resolve a target and measurable goal. Ask only when speed, memory, and size would choose materially different work.
2. Measure a baseline using existing benchmarks or a bounded appropriate measurement. If no measurement is possible, label the result an unmeasured static complexity review.
3. Rank candidates by expected impact divided by effort.
4. Do not rename public APIs, merge modules, change protocols, schemas, or folder structure. Such work requires an owner-approved Architecture Change Request.
5. Apply one change at a time, run tests, re-measure, and retain only a real improvement or an explicit size/readability gain. Revert neutral or regressive experiments.
6. In RWANG projects, append Benchmark and Optimize events.

Report baseline versus after, measurement method, retained changes, reverted attempts, and rejected ideas. State noise and uncertainty honestly.
